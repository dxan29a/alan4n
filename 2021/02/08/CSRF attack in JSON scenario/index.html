CSRF attack in JSON scenario

limitation factor

When the CSRF vulnerability point of the website uses the json data format, there are many restrictions on CSRF

  • The data sent needs to be in JSON format, it is difficult to construct and send JSON format data using HTML forms
  • Content-Type verification, using XMLHttpRequests will send OPTIONS pre-check requests to the server

Method 2

This method requires the use of Flash. When the server verifies the COntent-Type, it cannot use CSRF by simply sending text format data to construct JSON data. It requires Flash+PHP307 redirection combined to send data to the CSRF vulnerability point.

The specific process is that the Flash file requests a php page, and the php page performs a 307 redirect to the CSRF vulnerability point of the target website. Since 307 is a special redirect, he will also send the data received from the flash file to the target site to complete the CSRF attack.

First, we need to add the JSON data we want to send and the link to the php page in the [Flash file](SWF file). Use the FFDec tool to complete the editing of the Flash file.

Then add a PHP307 jump page on your own server

At this time, you only need to ask the user to request this Flash file.

There is also a simpler method. Someone wrote a more flexible and more reusable attack method on github, which also uses Flash to bypass. The difference is that instead of pre-editing Flash, Flash file variables are assigned according to the parameters in the user’s URL, and then 307 is triggered to pass JSON data. Link here.

方法2

这个方法需要使用Flash。当服务器对COntent-Type进行校验时，不能通过简单的发送text格式数据构造JSON数据来利用CSRF，需要Flash+PHP307重定向结合向CSRF漏洞点发送数据。
