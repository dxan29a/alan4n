Json web token(JWT)

发表于 2018-08-28 | 更新于 2019-08-28 | 分类于 CTF学习 | 阅读次数：

看到网鼎杯web的一道题mark一下JWT 全称是Json Web Token，用来代替session的一种验证方式 一般服务器辨别客户端身份的方式使用session 1、在客户端第一次登陆的时候向服务器发送用户名和密码。 2、服务器验证之后在当前对话（session）中写入信息（用户角色等）。 3、生成一个session_id发送给客户端，写入用户cookie。 4、以后客户每次请求session_id都会随着cookie发送给服务器。 5、服务器收到用户session_id之后找到前期保存的数据，得知用户身份。 但是这种session认证的方法扩展性不够好。当服务器多于一台而用户需要向多台服务器发送请求的时候，需要多台服务器共享一个session_id。一种解决方案是将session_id写入数据库或者别的持久层，这样做的好处是架构清晰，缺点是工程量大。另一种方法就是不用session，使用我们今天所要说的JWT。 JWT的原理是当服务器生成一个json对象发送给用户，这个json对象里包含服务器用于辨别客户端身份的所有必要信息 { "姓名": "张三", "角色": "管理员", "到期时间": "2018年7月1日0点0分" } 每次用户请求的时候都会向服务器发送这个json对象，服务器从而辨别客户端身份，服务器端就不再保存信息，变成了无状态，从而更好的实现拓展。为了防止客户端篡改JWT，服务器再发送的时候会加上签名。 在网鼎杯的web题中i_am_admin中,登陆页面抓包会发现JWT 标准的JWT数据格式如下 中间用两个 . 将数据分割为三个部分 Header（头部） Payload（负载） Signature（签名） Header部分是一个JSON对象，描述的是json元数据 { "alg": "HS256", //算法，默认是HMAC SHA256 "typ": "JWT" //表示这个令牌的（token）类型为JWT } Payload部分包含了辨别客户端身份的必要信息，官方规定为7个要素供选用，当然也可以自定义私有字段 iss (issuer)：签发人 exp (expiration time)：过期时间 sub (subject)：主题 aud (audience)：受众 nbf (Not Before)：生效时间 iat (Issued At)：签发时间 jti (JWT ID)：编号 Signature部分是对前两部分的签名，防止篡改，首先制定一个密钥，这个密钥只有服务器才知道，使用Header中指定的算法生成签名，生成之后将三部分用点拼接Header.Payload.Signature发送给客户端。 JWT默认是不加密的，但也可以在生成后用密钥将三个部分再加密一次 JWT最大的缺点是因为服务器不再保存session状态，所以无法在中途废除一个token或者更改一个token的权限，除非服务器部署其他逻辑，也就是说一旦JWT签发，在到期之前就会始终保持有效。 最后mark一个网站可以根据secret密钥生成加密后的JWT https://jwt.io/ 参考链接： http://www.ruanyifeng.com/blog/2018/07/json_web_token-tutorial.html
